# Backend OSS Analyzers Spec

- Status: proposed
- Date: 2026-03-30

## Purpose

Define the implementation scope for a public, MIT-licensed Roslyn analyzer package that can replace the temporary `Menees.Analyzers` dependency currently used by this repository.

This document is meant to be handed directly to an implementation agent.

## Current Temporary State

- The repository currently uses `Menees.Analyzers` as a temporary build-only guard.
- It is wired through `Directory.Build.props`.
- Thresholds live in `Menees.Analyzers.Settings.xml`.
- Severities live in `.editorconfig`.

The target outcome of this spec is:

1. build our own analyzer package
2. publish it as OSS under MIT
3. use it in this repository with `PrivateAssets="all"`
4. remove `Menees.Analyzers` once parity is good enough

## Why Build Our Own Package

- Minimize long-term trust in third-party analyzer dependencies.
- Keep guard behavior fully transparent and repository-driven.
- Add repo-specific architecture rules that general-purpose analyzers do not model well.
- Still keep the final solution reusable outside this repository.

## Non-Goals

- Do not build a full Sonar replacement.
- Do not build style analyzers that duplicate `dotnet format`, StyleCop, or generic naming rules.
- Do not add runtime dependencies to application projects.
- Do not require a custom XML settings format for V1 if `.editorconfig` can carry the configuration.
- Do not try to ship every imaginable Clean Architecture rule in the first release.

## Product Requirements

The package must:

- be public OSS under MIT
- be consumable as a standard NuGet analyzer package
- run in `dotnet build`, `dotnet test`, Rider, and Visual Studio
- have zero runtime impact on the application
- be configurable through `.editorconfig`
- ignore generated code by default
- produce stable, documented diagnostic IDs
- keep false positives low enough that warnings stay actionable

## Preferred Package Shape

Working structure:

1. `src/<PackageName>.Analyzers/`
2. `tests/<PackageName>.Analyzers.Tests/`
3. `docs/`
4. `README.md`
5. `LICENSE`

Preferred implementation stack:

- Roslyn analyzer package
- xUnit analyzer tests
- `Microsoft.CodeAnalysis.CSharp.Analyzer.Testing`
- one analyzer per focused rule
- code fixes only where the fix is obvious and low-risk

## Configuration Contract

Use `.editorconfig` as the primary configuration surface.

Severity:

- use standard Roslyn severity configuration:
  - `dotnet_diagnostic.<ID>.severity = none|silent|suggestion|warning|error`

Custom options:

- prefer `AnalyzerConfigOptionsProvider`
- support glob-scoped config, for example:
  - `[src/**/*.cs]`
  - `[tests/**/*.cs]`
  - `[tools/**/*.cs]`

Recommended option naming:

- `<package_prefix>.max_file_lines`
- `<package_prefix>.max_method_lines`
- `<package_prefix>.max_accessor_lines`
- `<package_prefix>.max_endpoint_method_lines`
- `<package_prefix>.max_route_lambda_lines`
- `<package_prefix>.allow_missing_cancellation_for_tests`

Do not require a custom XML settings file for V1.

## Rule Set Strategy

Build the analyzer package in two layers:

1. `Core maintainability guards`
2. `Backend architecture guards`

The core guards replace the current temporary analyzer.
The backend architecture guards are repository-informed and are the main reason to own the package.

## V1 Required Rules

These rules are the minimum useful replacement for the current temporary setup.

### `BGA001` File Too Long

Goal:

- flag oversized `.cs` files before they become god files

Requirements:

- configurable max lines via `.editorconfig`
- count physical lines, but allow ignoring blank lines and comments if the implementation agent can do so reliably
- ignore generated files:
  - `*.g.cs`
  - `*.generated.cs`
  - `*.designer.cs`
  - files containing `<auto-generated`

Initial default targets for this repository:

- `src/**/*.cs`: `800`
- `tests/**/*.cs`: `1000`
- `tools/**/*.cs`: `800`

Notes:

- this rule is a hard backstop, not the desired design target
- docs and AGENTS may still recommend earlier extraction at `~400` or `~500`

### `BGA002` Method Too Long

Goal:

- surface methods that are becoming orchestration blobs

Requirements:

- configurable max lines via `.editorconfig`
- work for methods, local functions, constructors, and maybe lambdas only if lambda analysis is cheap and precise
- ignore trivial expression-bodied members

Initial default targets:

- `src/**/*.cs`: `100`
- `tests/**/*.cs`: `140`
- `tools/**/*.cs`: `100`

### `BGA003` Accessor Too Long

Goal:

- catch hidden complexity inside property getters/setters

Requirements:

- configurable max lines
- analyze `get`, `set`, `init`
- ignore expression-bodied accessors

Initial default target:

- all backend `.cs`: `60`

### `BGA004` Async Method Should Expose Cancellation

Goal:

- encourage cancellable async flows in services and orchestration code

Requirements:

- trigger on `Task`, `Task<T>`, `ValueTask`, `ValueTask<T>` methods
- support exclusions to avoid noise:
  - event handlers
  - overrides where the base signature cannot be changed
  - interface implementations where the interface cannot be changed
  - test methods
  - entry points like `Main`
- allow a method to satisfy the rule either by:
  - taking a `CancellationToken`
  - taking a parameter with a public `CancellationToken` property
  - or another documented equivalent recognized by config

Initial severity suggestion:

- `suggestion`

### `BGA005` Prefer `TryGetValue`

Goal:

- catch `ContainsKey` + indexer double lookup patterns

Requirements:

- detect common `Dictionary` and `IReadOnlyDictionary` patterns
- avoid false positives when code path semantics would change
- no code fix required for V1, but acceptable if safe

Initial severity suggestion:

- `suggestion`

## V2 Repository-Informed Rules

These are the rules that make the package genuinely worth owning.
They should be designed now, but can ship after V1.

### `BGA100` Endpoint Mapping Must Stay Thin

Goal:

- keep endpoint-mapping modules transport-focused

Target surface:

- files under `Endpoints/`
- methods named like `Map*Endpoints`

Diagnostics to consider:

- mapping method exceeds configured max lines
- route lambda exceeds configured max lines
- endpoint mapping directly uses infrastructure APIs such as:
  - `System.IO`
  - `System.Diagnostics.Process`
  - `Microsoft.Data.Sqlite`
  - raw `HttpClient`

Expected intent:

- endpoint files should wire routes, authorize, shape transport contracts, and delegate to services

### `BGA101` Options And Contracts Must Stay Data-Only

Goal:

- keep options/DTO types free of hidden workflow logic

Target surface:

- files under `Options/`
- types named `*Options`
- DTO/contracts folders if the agent can identify them safely

Diagnostics to consider:

- non-trivial methods inside option/contract types
- direct IO, HTTP, process, or database calls from those types
- service locator or DI usage in those types

### `BGA102` Multiline Copy Should Not Use `Environment.NewLine` Stitching

Goal:

- discourage unreadable backend response/copy assembly

Diagnostics to consider:

- repeated string concatenation or interpolation chains with multiple `Environment.NewLine`
- recommend raw string literals or dedicated formatter helpers instead

### `BGA103` Endpoint Orchestrator Should Not Mix Too Many Responsibility Clusters

Goal:

- surface classes that combine transport mapping, normalization, persistence, IO, and policy in one place

Note:

- this is intentionally hard
- do not implement in V1 unless the implementation agent can design a low-noise heuristic

Possible heuristics:

- too many constructor dependencies from unrelated namespaces
- simultaneous direct usage of endpoint primitives plus filesystem/process/database APIs

### `BGA104` Large Backend File In Hotspot Namespace

Goal:

- allow stricter thresholds in the most sensitive folders

Examples:

- `Services/Jobs/Execution/`
- `Services/Jobs/Collaboration/`
- `Endpoints/`

This may be implemented simply as better `.editorconfig` scoping rather than as a separate analyzer.

## Rule Priorities

Recommended delivery order:

1. `BGA001`
2. `BGA002`
3. `BGA003`
4. `BGA004`
5. `BGA005`
6. `BGA100`
7. `BGA101`
8. `BGA102`

Do not start with the most “clever” architecture rule.
Start by replacing the current temporary value with stable parity.

## False Positive Policy

The implementation agent must optimize for low-noise diagnostics.

Rules should prefer:

- missing some edge cases over spamming common legitimate patterns
- explicit exclusions over magical heuristics
- conservative analysis in tests
- clear messages that explain the actual issue and the preferred alternative

Every rule should document:

- what it matches
- what it intentionally ignores
- known safe exclusions

## Required Documentation

The analyzer package must ship with:

1. a `README.md` with rule table
2. one short doc section per rule
3. `.editorconfig` examples
4. example suppressions
5. migration instructions for replacing `Menees.Analyzers` in this repository

## Acceptance Criteria For V1

V1 is acceptable when:

1. the package builds and tests cleanly
2. `BGA001` to `BGA005` exist and are documented
3. the package can be wired into this repository through `Directory.Build.props`
4. configuration works through `.editorconfig`
5. generated code is ignored
6. `dotnet build AgentServiceTemplate.sln --nologo` succeeds with the package enabled
7. the warning set is close enough to current `Menees.Analyzers` coverage that the repo can switch without losing visibility on oversized backend files and methods

## Migration Plan For This Repository

Phase 1:

- keep `Menees.Analyzers` as the temporary dependency
- implement the custom OSS analyzer package outside or beside the repo

Phase 2:

- add the custom package to `Directory.Build.props`
- mirror the current thresholds in `.editorconfig`
- run `dotnet build`
- compare diagnostics against the current temporary baseline

Phase 3:

- remove `Menees.Analyzers`
- delete `Menees.Analyzers.Settings.xml`
- update:
  - `docs/architecture.md`
  - `src/AGENTS.md`
  - `tests/AGENTS.md`
  - `.codex/skills/backend-clean-architecture-guard/SKILL.md`
  - `.codex/skills/backend-clean-architecture-guard/references/*`

## Nice-To-Haves

- code fix for `ContainsKey` + indexer -> `TryGetValue`
- SARIF-friendly metadata
- help links per diagnostic
- analyzer benchmarks or performance tests
- sample GitHub Actions workflow for OSS consumers

## Explicit Recommendation To The Implementation Agent

- Ship V1 small and boring.
- Replace current maintainability parity first.
- Add repository-specific architecture rules only after the package already works cleanly in a real solution.
- Prefer `.editorconfig` over custom config files unless Roslyn limitations force a second config surface.
