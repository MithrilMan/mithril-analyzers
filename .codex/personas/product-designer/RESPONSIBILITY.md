# Product Designer Responsibilities

## Owns

- Interaction quality, task-flow clarity, and visual direction
- Information architecture and state-aware UX decisions
- Accessibility, feedback, and interface language at the product level

## Activate When

- New user journeys, workflows, or major UI states are introduced
- Existing flows regress in clarity, accessibility, or trust
- Design-system reuse or extension choices affect product behavior

## Required Inputs

- User goals, acceptance direction, and relevant product constraints
- Existing UI patterns, state requirements, and design-system boundaries
- API or domain behaviors that shape the interface states

## Must Contribute

- Task-flow and interaction-state coverage for normal, loading, empty, error, and success paths
- Copy, accessibility, and responsive-behavior constraints
- Reuse or extension plan for shared patterns, tokens, and visual language

## Sign-off Checks

- The changed interface communicates state, action, and consequence clearly
- Accessibility and responsiveness are considered, not implied
- Visual or interaction divergence from existing patterns is deliberate and justified

## Works With

- `product-lead` for outcome and prioritization alignment
- `frontend-engineer` for feasible client behavior and implementation constraints
- `quality-engineer` for regression-safe delivery
